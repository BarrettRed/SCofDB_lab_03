### Запрос №1 — Заказы пользователя с сортировкой
```sql
SELECT id, status, total_amount, created_at
FROM orders
WHERE user_id = (SELECT id FROM users ORDER BY id LIMIT 1)
ORDER BY created_at DESC
LIMIT 10;
```

Ключевые узлы плана:
- `Seq Scan on orders` — полный последовательный перебор 100 000 строк
- `Rows Removed by Filter: 99986` — отфильтровано почти всё
- `Sort Key: orders.created_at DESC` — сортировка после фильтрации
- `Time: 9.556 ms`

Почему медленный: нет индекса по `user_id`, поэтому PostgreSQL читает всю таблицу orders.

### Запрос №2 — Фильтрация по статусу и диапазону дат
```sql
SELECT id, user_id, total_amount, created_at
FROM orders
WHERE status = 'paid'
  AND created_at >= '2024-12-01'
  AND created_at < '2024-12-08';
```

Ключевые узлы плана:
- `Seq Scan on orders` — полный перебор 100 000 строк
- `Rows Removed by Filter: 99546`
- `Time: 7.537 ms`

Почему медленный: нет составного индекса по `(status, created_at)`, поэтому приходится читать всю таблицу и применять оба условия фильтра последовательно.

### Запрос №3 — JOIN + GROUP BY (топ пользователей по сумме)
```sql
SELECT o.id, SUM(oi.price * oi.quantity) AS total_spent
FROM orders o
JOIN order_items oi ON oi.order_id = o.id
WHERE o.user_id = (SELECT id FROM users ORDER BY id OFFSET 5 LIMIT 1)
GROUP BY o.id
ORDER BY total_spent DESC
LIMIT 10;
```

Ключевые узлы плана:
- `Seq Scan on orders` - полное сканирование таблицы заказов (100 000 строк).
- `Rows Removed by Filter: 99994` - отфильтровано почти всё, оставив только 6 строк, но пришлось прочитать всю таблицу.
- `Seq Scan on order_items` - полное сканирование таблицы позиций заказов (400 000 строк).
- `Hash Join` - соединение результатов двух последовательных сканирований, так как нет индексов.
- `Sort & GroupAggregate` - сортировка для группировки по o.id и итоговая сортировка по total_spent.
- `Time: 52.489 ms`

Почему медленный: 
- Отсутствие индекса на orders.user_id:
Чтобы найти заказы одного пользователя, PostgreSQL выполняет Seq Scan (последовательное чтение) всей таблицы orders (100к строк). 

- Отсутствие (или неиспользование) индекса на order_items.order_id:
Для того чтобы найти товары, относящиеся к 6 найденным заказам, планировщик выполняет Seq Scan таблицы order_items на 400 000 строк.

- Неэффективный тип соединения (Hash Join):
Поскольку подходящих индексов нет, планировщик вынужден выкачивать обе таблицы целиком и соединять их через хеш-таблицу.

## 5. Партиционирование `orders` по дате
### 5.1 Выбранная стратегия

### 5.2 Реализация
_TODO: Кратко опишите, как реализовали партиционирование и перенос данных._
1. Исходная таблица `orders` переименована в `orders_partitioned`

4. Данные перенесены: `INSERT INTO orders SELECT * FROM orders_partitioned`

6. Созданы индексы на партиционированной таблице по `(user_id, created_at DESC)` и `(status, created_at)`

## 7. Что удалось исправить
_TODO: Перечислите конкретные проблемы и как именно вы их исправили._

- Устранение полных сканирований (Seq Scan): В запросах Q1 и Q2 медленное чтение всей таблицы (100к строк) заменено на эффективный Index Scan.

- Оптимизация JOIN (Q3): За счет индекса на внешнем ключе order_items.order_id база перешла от тяжелого Hash Join к быстрому Nested Loop, что дало самый большой прирост производительности (в 75 раз).

- Исключение лишней сортировки: В Q1 использование составного индекса (user_id, created_at DESC) позволило базе получать данные уже в нужном порядке, исключив этап Sort из плана выполнения.

## 8. Что не удалось исправить только индексами
_TODO: Перечислите случаи, где индексы не помогли, и объясните почему._

Подсказки:
- full scan при высокой доле выбираемых строк;
- тяжёлые `GROUP BY`/`ORDER BY` на большом объёме;
- необходимость переписывания запроса или pre-aggregation.

- Запросы с низкой селективностью: Индексы практически бесполезны для фильтрации по статусам (например, status = 'paid'), если этот статус имеет большинство записей в таблице. В таких случаях база всё равно выберет Seq Scan.

## 9. Выводы
_TODO: Сформулируйте 3-5 практических выводов по диагностике и оптимизации PostgreSQL._
Селективность — ключевой фактор: Индекс эффективен только тогда, когда он помогает отсеять более 90-95% данных. Для массовых отчетов индексы на фильтрах работают плохо.

JOIN без индексов: Отсутствие индекса на внешнем ключе (Foreign Key) в таблице связей (как order_items) приводит к катастрофическому падению скорости даже при малом количестве результирующих строк.

Партиционирование не всегда дает мгновенный прирост скорости в «лабораторных» условиях на 100к строк, но является обязательным для поддержания производительности при росте базы.
