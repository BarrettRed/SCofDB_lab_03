## 2. Найденные медленные запросы (до оптимизации)

### Запрос №1 (Фильтрация по юзеру + сумме + сортировка)
1. 
```sql
EXPLAIN (ANALYZE, BUFFERS)
SELECT id, status, total_amount, created_at
FROM orders
WHERE user_id = (SELECT id FROM users ORDER BY id LIMIT 1) AND total_amount > 1500
ORDER BY created_at DESC
LIMIT 10;
```
2. **План:** Seq Scan on orders -> Sort -> Limit
3. `Execution Time`: 8.946 ms
4. **Почему медленный:** Seq Scan читает все 100 000 строк для поиска заказов одного юзера, а затем тратит память на явную сортировку (Sort) по дате.

### Запрос №2 (Статус + Диапазон дат)
1. 
```sql
EXPLAIN (ANALYZE, BUFFERS)
SELECT id, user_id, total_amount, created_at
FROM orders
WHERE status = 'paid'
  AND created_at >= '2024-12-01'
  AND created_at < '2024-12-08';
```
2. **План:** Seq Scan on orders
3. `Execution Time`: 7.058 ms
4. **Почему медленный:** Seq Scan читает все 100 000 строк и проверяет каждую. Несмотря на то, что `paid` занимает 50% таблицы, узкий диапазон дат в 1 неделю сильно ограничивает выборку, но база все равно читает всю таблицу.

### Запрос №3 (Подсчет чеков для конкретного пользователя)
1. 
```sql
EXPLAIN (ANALYZE, BUFFERS)
SELECT o.id, SUM(oi.price * oi.quantity) AS total_spent
FROM orders o
JOIN order_items oi ON oi.order_id = o.id
WHERE o.user_id = (SELECT id FROM users ORDER BY id OFFSET 5 LIMIT 1)
GROUP BY o.id
ORDER BY total_spent DESC
LIMIT 10;
```
2. **План:** Parallel Hash Join -> Sort -> GroupAggregate -> Sort -> Limit
3. `Execution Time`: 54.183 ms
4. **Почему медленный:** База данных делает Seq Scan по `orders` для поиска заказов одного пользователя, а затем `Parallel Seq Scan` по всей таблице `order_items` (400k строк), чтобы найти товары для этих нескольких заказов. Это крайне неэффективно.

## 3. Добавленные индексы и обоснование типа

### Индекс №1
- **SQL:** `CREATE INDEX idx_orders_user_created ON orders USING BTREE (user_id, created_at DESC);`
- **Какой запрос ускоряет:** Q1, Q3
- **Почему выбран:** Композитный B-Tree индекс. Сначала фильтрует по `user_id`, а сортировка `created_at DESC` позволяет планировщику избежать узла `Sort` в Q1 и просто прочитать первые 10 подходящих записей с конца индекса. Условие `total_amount > 1500` проверяется как фильтр при чтении индекса. В Q3 ускоряет поиск заказов конкретного пользователя.

### Индекс №2
- **SQL:** `CREATE INDEX idx_orders_status_date ON orders USING BTREE (status, created_at);`
- **Какой запрос ускоряет:** Q2
- **Почему выбран:** Композитный B-Tree индекс. Статус `paid` имеет плохую селективность (50%), но в комбинации с датой работает отлично. Сначала идет поиск по точному равенству (status), затем по диапазону (created_at).

### Индекс №3
- **SQL:** `CREATE INDEX idx_order_items_order_id ON order_items USING BTREE (order_id);`
- **Какой запрос ускоряет:** Q3
- **Почему выбран:** B-Tree идеален для Foreign Key. Ускоряет процесс JOIN между `orders` и `order_items`, заменяя полное сканирование таблицы товаров (`Parallel Seq Scan`) на очень быстрый точечный поиск `Bitmap Index Scan` для конкретных заказов пользователя (`Nested Loop`).

## 5. Партиционирование `orders` по дате
### 5.1 Выбранная стратегия
RANGE по кварталам, т.к. оптимальное кол-во партиций - 8 (по ~12.5% данных). Разбиение по месяцам дало бы слишком много партиций (24), а по годам - слишком мало (2), что снизило бы эффективность Partition Pruning.

### 5.2 Реализация
1. Создана таблица `orders_partitioned` с `PARTITION BY RANGE (created_at)`
2. Созданы 8 партиций по кварталам: `orders_2024_q1` ... `orders_2025_q4`
3. Создана `DEFAULT` партиция для строк вне диапазонов.
4. Перенесены данные `INSERT INTO orders_partitioned SELECT * FROM orders;` и созданы индексы на партиционированной таблице.

### 5.3 Проверка эффекта
Запрос Q5 (аналог Q2) сработал только по одной партиции (`orders_2024_q4`), что подтверждает работу Partition Pruning.

## 7. Что удалось исправить

- **Q1:** Избавились от полного перебора. Индекс `idx_orders_user_created` позволил найти нужные строки моментально (Bitmap Index Scan).
- **Q2:** Ускорение за счет составного индекса. Партиционирование дало дополнительный буст (Partition Pruning), так как запрос сканировал только партицию 4 квартала 2024 года.
- **Q3:** Тяжелый `Parallel Hash Join` (соединение двух полных таблиц) был заменен на элегантный `Nested Loop`. База находит несколько заказов юзера, и для каждого заказа точечно лезет в индекс `idx_order_items_order_id`. Время упало с 54.183 мс до 0.776 мс. После партиционирования время слегка возросло до 1.113 мс из-за необходимости обходить все партиции (так как нет фильтра по дате).

## 8. Что не удалось исправить только индексами

- **Q1 с партиционированием:** Теоретически, так как в Q1 нет условия по `created_at` (ключ партицирования), база данных вынуждена делать Index Scan по всем 8 партициям. В "холодном" состоянии это может замедлить запрос. Однако на практике время даже улучшилось с 3.560 ms (после индексов) до 1.568 ms (после партиционирования). Это произошло исключительно за счет **прогрева кэша** (shared_buffers): к моменту запуска скрипта партиционирования нужные страницы данных и индексов уже были загружены в оперативную память.

### Почему тайминги меняются при каждом запуске?
Разброс во времени выполнения — это нормальное явление для баз данных, которое во многом связано с состоянием кэша.
При самом первом запуске запроса базе данных приходится читать данные с физического жесткого диска. В последующие разы блоки данных оседают в кэше PostgreSQL (`shared_buffers`) и операционной системы. Именно поэтому в Q1 эффект кэширования оказался сильнее теоретических накладных расходов от обхода 8 партиций (время уменьшилось с 3.5 мс до 1.5 мс).

## 9. Выводы

1. **Композитные индексы избавляют от сортировок.** Порядок колонок `(user_id, created_at DESC)` критически важен. Если бы сортировка в индексе была `ASC`, базе пришлось бы переворачивать данные в памяти.
2. **Селективность решает.** Поиск по `status = 'paid'` без даты привел бы к сканированию полтаблицы даже с индексом, потому что половина таблицы имеет такой статус. Но добавление даты в составной индекс спасает ситуацию.
3. **Индексы для Foreign Key обязательны.** Без индекса на `order_id` в таблице `order_items` база данных вынуждена прибегать к тяжелому `Hash Join` и сканировать сотни тысяч записей, даже если мы ищем товары только для одного пользователя.
4. **Partition Pruning работает не всегда.** Партиционирование помогает только тогда, когда в `WHERE` есть условие по ключу партицирования (дата). Без него база вынуждена обходить все партиции, что добавляет накладные расходы.
